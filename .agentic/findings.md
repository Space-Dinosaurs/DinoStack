## Architect plans cannot gate on preconditions the installing plan itself violates

**Pattern:** When a rule references "the Architect plan" as a gate condition (e.g. "the Architect plan must have empty open questions"), the plan installing that rule must itself satisfy the gate. Otherwise the first real invocation of the rule is blocked by the very plan that created it.
**Where it bites:** Multi-phase methodology evolution where each phase's architect output feeds into rules that constrain future architect outputs. The circularity is invisible until a Skeptic traces the gate back to the installing plan.
**How to avoid:** Before shipping a rule that references "the Architect plan" as a precondition, re-read the current plan against that precondition. If the current plan violates it, resolve the violation (close open questions, rewrite ambiguous sections) before the rule ships. Treat "this rule cannot apply to the plan installing it" as an automatic Skeptic-caliber Critical.
**Example:** 2026-04-15, Phase 1.3 v1 of the harness-engineering implementation - the architect's plan carried 4 open questions AND proposed a rule requiring the Architect plan's Open Questions section to be empty. The first real use would be blocked by the plan that created the rule.

## tool_scope and output_paths are behavioral gates, not harness-enforced barriers

**Pattern:** Command docs and architect plans that describe proposal-only analyst Workers as "physically gated" against writing to protected paths overstate the safety model. The NLH execution contract's `tool_scope` and `output_paths` fields are documentation the Worker brief carries - they do not override harness-level tool grants. The actual gate is (a) Worker brief compliance and (b) the conductor rejecting unexpected diffs before the next step.
**Where it bites:** Safety claims in command specs imply the harness will refuse a write the Worker brief forbade, when in reality a non-compliant Worker can still produce the write and only the conductor's diff review catches it. Overstated safety language encourages skipping the diff review.
**How to avoid:** Always describe these fields as behavioral gates. Standard phrasing: "the analyst is instructed not to write to `content/`, and any violation would surface as a diff the conductor rejects before the next step." Never write "physically cannot edit" or "the harness will block." When reviewing a command spec, flag any language implying harness enforcement of a brief-level constraint as a Major.
**Example:** 2026-04-15, Phase 1.2 Skeptic review of `/prune-harness` - the original plan claimed the mechanism "physically cannot delete without a second deliberate user action." Revised to "the analyst is instructed not to write to `content/`, and any violation would surface as a diff the conductor rejects." This phrasing is now house style across both `/prune-harness` and `/representation-audit`.

## Commit-before-Skeptic is a tempting optimization that violates the Irreversible-changes rule

**Pattern:** Architect plans that argue for removing the post-impl Skeptic on "tight" tasks will often propose committing the fix first and reviewing after. This directly violates `skeptic-protocol.md`'s Irreversible-changes rule ("Workers must not apply irreversible changes before the primary agent has confirmed Skeptic sign-off"). A rationalization table row documents the rationalization as failing - it does NOT grant an exception to the rule.
**Where it bites:** Any optimization targeting Skeptic cost on well-specified tasks. The designer sees "the tests pass so Skeptic is redundant" as self-evident; the protocol does not agree, because a Worker that misreports its tests or commits a latent bug still lands broken state in git history before any independent review. This is precisely the failure mode the Irreversible-changes rule exists to prevent.
**How to avoid:** If commit-before-sign-off is genuinely needed in a new context, amend the Irreversible-changes paragraph in `skeptic-protocol.md` with a new explicit carve-out naming the sub-path, its preconditions, and the revert-to-default behavior on verification failure. Do NOT add a rationalization row and assume the exception is granted - rationalization rows document failure modes, they do not create permissions. An alternative that always works: have the Worker return the uncommitted diff to the conductor, which then commits after verifying the output.
**Example:** 2026-04-15, Phase 1.3 v1 of the tight-fix path - the original design had the Worker commit after local tests passed and fall back to a "retroactive Skeptic" on the committed diff. Skeptic flagged this as Critical because the Irreversible-changes rule was not amended. Phase 1.3 v2 resolved it by amending the rule with an explicit tight-fix carve-out AND keeping the Worker's commit conditional on in-Worker test pass, so no irreversible change happens before the gate.
